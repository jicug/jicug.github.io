### 事前雑談：IBM Watsonx.ai（IBM Bob）の料金プランについて

- IBM Bobがバージョン2.0にアップデートされた。旧バージョンのプロジェクトにはマイグレーション確認ダイアログが表示される
- IBM Bobの課金単位は「ボブコイン」と呼ばれる独自単位
  - Proプラン：約3,187円（税別）／月、40ボブコイン付与
  - Pro+プラン：約9,562円（税別）／月、160ボブコイン付与
  - Ultraプラン：約31,900円（税別）／月、200ドル相当
- 超過使用時は1ボブコインあたり0.5ドルの追加課金が発生するため、ガンガン使う場合は上位プランの方がコスパが良い
- 個人利用ではProまたはPro+が現実的。SIerや業務委託での本格利用にはUltraが適切と考えられる
- トークンのリセットは契約日から30〜31日サイクル（アニバーサリーデート）

### 発表①：Bob IDE で IBM Cloud CLI 経由 Code Engine を動かしてみて動作させるまでやってみるよもやま話（田中氏）

- **概要**：フロントエンドエンジニアがIBM Bobと協力し、GUIのポチポチ操作をCLIコマンドで代替してCode Engineへのデプロイを実現した体験談
- **取り組み内容**
  - 「経緯管理システム」（ドックスレター＝申し送り文書、ノート）をIBM Bob自身に書かせ、セッション切断時の文脈引き継ぎに活用
  - GitLeaksなどを用いたセキュリティチェック（コミット時の機密情報漏洩防止）を導入
  - IBM Cloud CLIとCode Engineプラグインを組み合わせ、ビルド・デプロイをコマンドで実行
  - Bootstrapを追加してUIを更新し、再デプロイも成功
- **ボブコイン消費の振り返り**
  - 環境設定に約3.5ボブコイン、ソースコード読み込みに約10ボブコイン（月40コインの約1/4）
  - 後半は雪だるま式に消費が増加。一セッションで多くのタスクを並行させると文脈が広がりトークン消費が増える
  - 単機能（Bootstrapのみ等）に絞ったセッションはコイン消費が少なかった
- **学び・気づき**
  - タスクを細かく区切ってセッションを分離するとコスパが良い
  - IBM BobはIBM Cloudのヘルプ情報を内部に持っており、CLIコマンドを適切に生成してくれた
  - コンテナレジストリの容量超過（80%超）をIBM Bobが検知し、自動でクリーンアップしてくれた
  - インフラに不慣れなエンジニアでも、IBM Bobを「事情を知っているパートナー」として活用することで開発に集中できる

### 発表②：AIでアプリは作れる。ならば、AIは作ったアプリをクラウドにデプロイすることはできるのか（鹿野氏）

- **概要**：オブザーバビリティ教育用の昭和レトロクイズアプリ（Node.js＋MySQL構成）を、IBM Cloudのサーバーレス環境へ移行した体験談
- **選定したIBM Cloudの無料サービス**
  - Webサイト：IBM Cloud Object Storage（COS）
  - アプリ実行：Code Engine（月10万秒＝約28vCPU時間まで無料）
  - データベース：当初Db2を選定
- **Db2断念の経緯**
  - Db2の無料（Liteプラン）は北米（us-south）と英国（eu-gb）の2リージョンのみ対応で、東京リージョンでは提供されていない
  - さらにTerraformによる自動構築が非対応であることが判明し、Db2を断念
  - 代替としてCloudant（NoSQL）を採用。日本リージョン対応・完全無料
- **デプロイ手法の比較検討**
  - 手動（管理ポータル）、MCPサーバー、IBM Cloud CLI、Terraformを比較
  - Terraformが最も優位と判断し採用。MCPサーバーも将来的に試したい
- **Terraformによるデプロイの実施**
  - IBM BobがTerraformの定義ファイルを生成し、アーキテクチャ図・デプロイガイドライン・トラブルシューティングドキュメントも自動作成
  - 発生したエラー：プロジェクト名の重複、IAMロールの形式不一致、Dockerイメージの消失など
  - 手作業が必要だった箇所：IBM Cloud APIキー（IBM Cloudポータルで発行）、IBM Cloud COS APIキー（`terraform apply`実行後、ポータルのCOSサービス資格情報から発行）、CLIのインストールなど
  - 最終的にCode Engine・VPC・セキュリティグループ・Object Storageなどのリソースが自動構築され、アプリの動作確認に成功
- **ライブデモ（Terraform destroy）**
  - 会の最後にTerraform destroyを実行し、構築したリソースをほぼ自動削除
  - VPC・コンテナイメージレジストリ・セキュリティグループの3つは手作業での削除が必要と確認
- **今後の課題**
  - Terraformコードの内容精査
  - MCPサーバーとIBM Bobの連携検証
  - クラウド環境でのオブザーバビリティ勉強会の開催

### 発表③：Claude と Bob で業務アプリを作る（西川氏）

- **概要**：仕様駆動開発（SDD：Spec-Driven Development）を活用し、ボブコインを節約しながら大学の学習証明ステッカー管理システムを開発した体験談
- **SDDとは**
  - 人間が仕様書を作成 → AIがコードを生成 → 人間がテスト・検証 → 仕様に追記するサイクル
  - 日本型SIer開発（要件定義→設計→開発→テスト）と構造的に近く、仕様書を書ける人材が鍵
- **コスト削減の工夫**
  - 仕様書・実装プロンプト集・プロジェクト指示書の3ファイルをClaudeの無料プランで作成
  - 作成した3ファイルをIBM Bobに読み込ませ、統合プランファイルを生成させてからコーディングを依頼
  - これにより仕様書作成フェーズのボブコイン消費をほぼゼロに抑え、コーディングのみで約20ボブコイン（従来比約10ボブコイン削減）
- **作成した3ファイルの内容**
  - 仕様書：システム概要・目的・設計原則・スコープ・ユースケース等（約30行の要件定義をIBM Bobに基本設計させると約700行に拡張）
  - 実装プロンプト集：DBスキーマ・テーブル定義・初期データ・マイグレーションスクリプト等
  - プロジェクト指示書：技術スタック・データモデル・命名規則等（SIerが協力会社に渡すドキュメントに相当）
- **成果**
  - ClaudeとIBM Bobでほぼ同一のデザイン・機能のアプリが再現でき、仕様書の再現性が確認された
  - Geminiではデザインがやや異なったがAnthropicのClaudeとIBM Bobは類似結果
- **議論・共有された知見**
  - 仕様書を先に作り込むことで、IBM Bobへの質問往復が減りコイン消費を大幅削減できる
  - 仕様書なしでコーディングを始めると「どうしましょう？」という質問が連発されコイン消費が急増する
  - 学生・専門学校生への指導方針として「まず仕様書（言葉で要件を表現すること）を書けるようになれ」を重視
  - 仕様書の品質がAIの出力品質を左右する点は、ある学会の場でも世代を超えて共感を得た話題

### 全体共通の知見・ディスカッション

- **セッション管理とトークン節約**
  - 一セッションで多くのタスクを並行させると文脈が広がりトークン消費が増加する
  - タスクを細かく区切り、セッションを分離することが有効
  - 申し送り文書（ドックスレター）をAI自身に書かせることでセッション間の文脈引き継ぎが効率化される
- **AIコーディングの品質管理**
  - AIが生成したコード・Terraformファイルは人間が目を通して監修する必要がある
  - IBM Bobは「安価な外注先・同僚」として活用するイメージが適切
  - 英語で指示するとトークン消費が減る可能性がある（日本語は文字数が多い分トークンを消費しやすい）
- **手作業が残る領域**
  - APIキーの生成・設定、CLIのインストール、一部のクラウドリソース削除などは現状手作業が必要
  - MCPサーバーを活用することで自動化できる範囲が広がる可能性があり、今後の検証課題
- **教育・人材育成への示唆**
  - AIを使いこなすには「技術用語を知っていること」と「要件を言語化できること」が重要
  - 仕様書を書ける人材の育成が、AI時代においても変わらず重要なスキルである

- 超過使用時は1ボブコインあたり0.5ドルの追加課金が発生するため、ガンガン使う場合は上位プランへの変更が合理的とのことだった。

## Bob IDE で IBM Cloud CLI 経由 Code Engine を動かしてみて動作させるまでやってみるよもやま話（田中氏）

- 田中正吾氏（フリーランスエンジニア）が、IBM Bobを使ってIBM CloudのCode Engine（サーバーレス環境）をCLIで操作・デプロイするまでの体験を発表した。
- AIとの作業で重要な「経緯を残す仕組み」として、ドックスレター（申し送り文書）とノート（知見記録）をマークダウン形式でAIに書かせることで、セッション切断後の文脈復元を効率化している。
- セキュリティ対策としてGitLeaksなどの自動コミット時セキュリティチェックを導入し、機密情報の誤コミットを防ぐ仕組みを整備した（環境設定に約3.5ボブコインを消費）。
- IBM Cloud CLIにCode Engine用プラグインを追加し、GUIのポチポチ操作ではなくコマンドベースでアプリ作成・ビルド・デプロイが実現でき、GitHubリポジトリとの同期も自動化できた。
- IBM BobはIBM Cloudのヘルプ情報を知識として持っており、コンテナレジストリの容量超過（80%超）を自動検知してクリーンアップを提案・実行するなど、インフラ管理の補助として機能した。
- 一つのセッション内で多くのタスクを処理すると文脈が広がりボブコイン消費が雪だるま式に増えるため、タスクを細かく区切ってセッションを分割することが節約の鍵だと学んだ。
- ブートストラップ導入など単体タスクに絞った場合はコイン消費が少なく、シンプルな依頼ほど高価なモデル（Claude等）より軽量なモデル（Granite LLM等）が動くため効率的であることが確認された。



## AIでアプリは作れる。ならば、AIは作ったアプリをクラウドにデプロイすることはできるのか（鹿野氏）

- 鹿野氏（株式会社東芝、マルチクラウド・ハイブリッドクラウド担当）が、オブザーバビリティ学習用の教材として昭和レトロクイズアプリ（Node.js＋MySQL構成）をAIで作成し、それをクラウド環境に移行した体験を発表した。
- 通常8週間かかる開発がAIを活用することで2日間で完成し、教育目的のアプリ開発に時間をかけずに済んだという実績が共有された。
- 無料枠を条件にIBM Cloudのサービスを選定し、Object Storage（COS）、Code Engine（月10万秒・28vCPU時間まで無料）、Db2（200MB無料）の3サービスを当初選んだ。
- IBM Bobが実行計画をステップごとに詳細に提示し、アーキテクチャ図やコスト試算（月55〜330回実行可能）まで自動生成してくれた点が他の生成AIと比較して優れていた。
- Db2が東京リージョンでは無料プランが存在しないことが判明し（無料プランは北米・英国の2リージョンのみ対応）、日米マルチリージョン構成への再設計を余儀なくされたが、IBM Bobのプランニング支援で人手よりも素早く対応できた。
- さらにDb2がTerraformでの自動作成に非対応であることが発覚し、最終的にNoSQLのCloudantに変更する判断を人間が下し、構成を完全に作り直した。
- デプロイ方法の選択肢として管理ポータル手作業・MCPサーバー・IBM Cloud CLI・Terraformを比較検討し、再現性・自動化の観点からTerraformが最も優位と判断した。
- TerraformやIBM Cloud CLIのパソコンへのインストールは人間の手作業が必須で、APIキーやシークレットキーの生成も自動化できず手作業が発生した。
- TerraformのApply実行中にプロジェクト名重複やIAMロール形式の古さなどのエラーが複数発生し、繰り返し修正・再実行しながらデプロイを進めた。
- コンテナイメージの手動削除によるエラーを含む複数のつまずきを経て、最終的にクラウドネイティブ環境（Object Storage＋Code Engine＋Cloudant）へのデプロイに成功した。
- Code Engineは60秒間アクセスがないと自動停止し課金が止まる仕組みを活用してコスト効率を高め、起動から約15秒でアクセス可能な状態になることを確認した。
- ライブデモとしてTerraform destroyコマンドを実行し、作成したリソース（Object Storage・Code Engine等）を分単位で削除できることを実演した（VPC・コンテナイメージレジストリ・セキュリティグループの3つのみ手作業での削除が必要）。
- 今後の課題として、Terraformの定義ファイルの読み解き・MCPサーバーとの連携、クラウド環境でのオブザーバビリティ勉強会の開催を予定している。



## Claude と Bob で業務アプリを作る（西川氏）

- 西川氏（大学教員・病院システムプロトタイピング等）が、仕様駆動開発（SDD: Spec-Driven Development）を用いてIBM Bobコストを削減しながら業務アプリを開発する手法を紹介した。
- SDDはまず人間が仕様書を作成しAIにコード生成させる手法で、従来のSIerにおける要件定義→外注開発の構造と本質的に同じであり、仕様書の品質がそのまま成果物の品質に直結することが強調された。
- 仕様書・実装プロンプト集・プロジェクト指示書の3ファイルをClaudeの無料プランで作成し、それをIBM Bobに読み込ませてプランファイルを統合生成させることで、仕様書作成工程のボブコイン消費を約10ボブコイン削減できた。
- 実際に大学での学習証明として物理ステッカーを管理するシステムを開発し、AnthropicのClaudeで作成したものとほぼ同じデザイン・機能のウェブアプリがIBM Bobでも再現でき、仕様書の再現性の高さが確認された。
- 同じ仕様書を使うとGeminiではデザインが若干異なったが、ClaudeとIBM BobはほぼIdenticalな出力となり、将来的に類似システムの派生開発や別フレームワークへの移植（Vue→React等）にも応用できることが示された。
- 要件定義書30行をIBM Bobに基本設計させると700行に自動拡張され、詳細な仕様がない状態でコーディングを開始すると質問が繰り返しボブコインが急速に消費されるため、事前の仕様書作り込みが節約の鍵であることが実例で示された。
- AIが生成したTerraformファイルにはトラブルシューティングガイドやデプロイ手順書まで自動作成されるが、品質担保のために人間によるレビューと手動修正が必要であり、IBM Bobは「安価な協力会社」として扱うのが実態に即しているとまとめた。



## 座談会：仕様書の重要性とAI活用の今後

- AI開発においていきなりコーディングを始めるのではなく、要件定義や仕様書を事前に作成することの重要性について全員が同意し、仕様書の品質がAIアウトプットの品質を決めるという認識で一致した。
- ある学会の例でも世代を超えたエンジニアが「仕様書を書ける人をいかに育てるか」というテーマで盛り上がっており、AI時代においても仕様書作成スキルの重要性は変わらないという見解が共有された。
- 専門学校・大学での教育として「自分がやりたいことを言葉にしないとAIは動かない」「技術用語を知ることで情報がAIに圧縮されて伝わりやすくなる」という指導方針が紹介された。
- AIが生成したDockerfileやTerraformコードは人間が書く場合より詳細でフォールバック処理なども含まれるが、読み解くのが難しいケースもあり、特に本番リリース時には人間によるレビューが不可欠である。
- IBM BobをはじめとするAIコーディングツールは「安価な協力会社（オフショア）」と同様の立場であり、英語で指示するとトークン効率が上がるなど、従来のSIerのオフショア開発と本質的に同じダイナミクスが働くと議論された。

- 近日中にCompassのイベントページを公開する予定であり、運営担当に申し込み受付の開放を依頼する場面があった。
